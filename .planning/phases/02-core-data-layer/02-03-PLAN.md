Create `src/memory/people.service.ts`:
```typescript
import { Inject, Injectable, Logger } from '@nestjs/common';
import nlp from 'compromise';
import { Pool } from 'pg';
import { PG_POOL } from '../database/database.constants.js';
import { PersonRow } from './memory.types.js';

@Injectable()
export class PeopleService {
  private readonly logger = new Logger(PeopleService.name);

  constructor(@Inject(PG_POOL) private readonly pool: Pool) {}

  detectNames(text: string): string[] {
    // compromise POS tagger returns deduplicated person name strings
    return (nlp(text).people().out('array') as string[]);
  }

  async lookupByNames(names: string[], userId: string): Promise<PersonRow[]> {
    // Guard: empty array would produce a vacuous ANY($2) and waste a round-trip
    if (names.length === 0) return [];

    const result = await this.pool.query<PersonRow>(
      `SELECT id, user_id, name, aliases, facts, created_at, updated_at
       FROM people
       WHERE user_id = $1
         AND (name = ANY($2::text[]) OR aliases && $2::text[])`,
      [userId, names],
    );
    return result.rows;
  }

  async upsertPerson(
    name: string,
    userId: string,
    facts: Record<string, unknown> = {},
  ): Promise<PersonRow> {
    const result = await this.pool.query<PersonRow>(
      `INSERT INTO people (user_id, name, facts)
       VALUES ($1, $2, $3)
       ON CONFLICT (user_id, name)
       DO UPDATE SET
         facts = people.facts || EXCLUDED.facts,
         updated_at = NOW()
       RETURNING id, user_id, name, aliases, facts, created_at, updated_at`,
      [userId, name, JSON.stringify(facts)],
    );
    return result.rows[0];
  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| text input → NLP (detectNames) | Raw conversation text from chat gateway — may contain adversarial input designed to confuse NLP |
| service method parameters → SQL | name, userId, facts arrive from callers; must never be interpolated into SQL |
| people.facts jsonb → application | JSONB returned from Postgres; typed as Record&lt;string, unknown&gt; — callers must narrow before using |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Elevation of Privilege | lookupByNames / cross-user people leak | mitigate | `WHERE user_id = $1` is the first condition in the query — structurally impossible to omit when userId is a required parameter. Unit test `'scopes query to user_id — WHERE user_id = $1'` asserts this. |
| T-02-03-02 | Injection | lookupByNames names parameter | mitigate | `$2::text[]` parameterization — pg handles array serialization; names are never string-interpolated. The `ANY($2::text[])` and `aliases && $2::text[]` patterns are fully parameterized. |
| T-02-03-03 | Injection | upsertPerson facts jsonb | mitigate | `JSON.stringify(facts)` serializes the object to a JSON string that pg inserts as a jsonb literal. No interpolation of facts keys/values into SQL. |
| T-02-03-04 | Elevation of Privilege | upsertPerson ON CONFLICT | accept | ON CONFLICT (user_id, name) — conflict resolution is scoped to the (user_id, name) unique constraint. A user cannot overwrite another user's person record because user_id = $1 is in the INSERT VALUES. |
| T-02-03-05 | Information Disclosure | detectNames return value | accept | detectNames processes text in-process with compromise NLP — no data leaves the server. Returns only extracted names to the service layer, not to the client. |
| T-02-03-06 | Denial of Service | lookupByNames large names array | accept | Reasonable v1 risk — names extracted from a single message by compromise will not exceed dozens. Production hardening (array length cap) is Phase 6 concern. |
</threat_model>
